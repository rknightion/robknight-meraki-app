    1. Install plugin, enable it.
    2. Open Apps -> Cisco Meraki -> Configuration (the in-app scene page). Paste

         per-sensor detail page with auto-hide-on-empty behaviours.

         alerts table.

       Configuration page) when the API key is missing.

1. LOCKED-IN ARCHITECTURE DECISIONS (do not revisit without strong reason)
================================================================================

1.1  Plugin ID namespace
     - v0.1 IDs: "rknightion-meraki-app"        (app plugin)
                 "rknightion-meraki-datasource" (nested data source)
     - Go module: github.com/robknight/grafana-meraki-plugin
     - At first SIGNED release, the user (Rob Knight) plans to migrate to his
       Grafana staff org "robknight". Expect a plugin-ID rename to
       "robknight-meraki-app" + "robknight-meraki-datasource" at that point.
       CHANGELOG already notes this.
     - The Go module path can stay as-is (github.com/robknight/...). Plugin ID
       and Go module path are independent.

1.2  Plugin type: App + nested DS + shared Go backend
     - The APP plugin has a Go backend (gpx_meraki). The app owns:
       • the Meraki API key (in app.secureJsonData.merakiApiKey)
       • the meraki.Client instance (rate limiter + cache + HTTP client)
       • resource endpoints: POST /resources/query, POST /resources/metricFind,
         GET /resources/ping
       • CheckHealth (calls GET /organizations)
     - The DATA SOURCE plugin is FRONTEND-ONLY. It has no backend binary.
       Its datasource.ts implements query() and metricFindQuery() by POSTing to
       the app's resource endpoints via getBackendSrv().fetch.
     - Rationale: one place to configure the API key, one Go binary to build,
       one rate limiter / cache shared across every query and every dashboard
       panel. This is the same pattern Grafana's Synthetic Monitoring app uses.

1.3  NO Prometheus / exporter dependency
     - The plugin does NOT depend on a Prometheus data source.
     - The plugin does NOT scrape, read, or import anything from the reference
       Python exporter at /Users/rob/repos/meraki-dashboard-exporter.
     - The exporter is READ-ONLY REFERENCE MATERIAL for:
       • Which Meraki endpoints exist and what they return
       • Per-endpoint rate-limit + pagination quirks
       • Dashboard panel layouts to reproduce as scenes
     - Every query from a scene goes directly to api.meraki.com via our Go
       backend — "Infinity-style" but with a typed query-kind contract.

1.4  Scenes everywhere. No provisioned static JSON dashboards.
     - Every observability flow is built as a @grafana/scenes page under
       src/pages/<Area>/. DO NOT add JSON dashboards under
       provisioning/dashboards/. If a v0.2+ phase wants a dashboard, build it
       as a SceneApp page.

1.5  Authentication
     - API key stored on the APP plugin in secureJsonData.merakiApiKey. Read
       server-side only via settings.DecryptedSecureJSONData. NEVER expose the
       key to the browser.
     - The nested DS has NO API-key field (ConfigEditor shows a message
       directing users to the app config page).

1.6  Rate limiting
     - Per-instance, per-org token bucket. 10 req/s default, burst 20, jitter
       ±10%. Implementation: pkg/meraki/ratelimit.go (token bucket with test
       hooks for Clock/Sleep/Rand).
     - Config knob: AppJsonData.sharedFraction (0 < x <= 1). Operators with N
       Grafana replicas set this to 1/N.
     - Distributed limiter is INTENTIONALLY deferred. Rely on the server-side
       429 + Retry-After as the natural coordinator across replicas.

1.7  Caching
     - In-memory TTL LRU, pkg/meraki/cache.go (wraps hashicorp/golang-lru/v2).
     - Key: sha256(orgID + path + sorted params) via CacheKey().
     - Per-kind TTLs (keep consistent with what's used in pkg/plugin/query/*):
         Organizations:          1h
         Networks:                15m
         Devices:                5m
         DeviceStatusOverview:   1m
         SensorReadingsLatest:    30s
         SensorReadingsHistory:   1m
     - No Redis, no cross-replica cache. Per-plugin-instance only in v0.1+.

1.8  Time-range handling
     - pkg/meraki/timerange.go defines EndpointTimeRange with MaxTimespan and
       AllowedResolutions, plus KnownEndpointRanges map.
     - Resolve() applies:
         • 60s freshness floor (subtract from `now` before computing t1)
         • Timespan clamp to MaxTimespan (emits Annotation if truncated)
         • Resolution quantization UPWARD to nearest allowed bucket
     - Current KnownEndpointRanges covers sensor history, apiRequests,
       wireless usage history, channel util history, uplinksLossAndLatency.
       ADD new entries when implementing v0.2+ timeseries endpoints.

1.9  Snapshot vs. timeseries conventions
     - Native timeseries only from: sensor readings history, wireless usage
       history, channel utilization history, uplinksLossAndLatency.
     - Snapshot endpoints (deviceStatusOverview, deviceStatuses, switch port
       statuses, VPN statuses) return point-in-time frames meant for Stat or
       Table viz. DO NOT try to synthesize timeseries from snapshots.
     - Future: a tiny opt-in 24h x 1min in-process ring buffer for
       deviceStatuses state transitions — NOT implemented yet; add in a v0.3+
       phase if users ask.

1.10 Frame-notice error pattern
     - Per-query errors become `data.Notice{Severity: Error, Text: ...}` on a
       placeholder frame. A single bad query does NOT kill the whole panel.
     - See pkg/plugin/query/dispatch.go for the wrapping logic. Keep this
       pattern for every new query kind.

1.11 URL-sync for scenes
     - Every scene page with drilldowns or tabs uses `routePath: 'path/*'`
       with a trailing `*`. Deep links must be bookmarkable.

2. CODE CONVENTIONS (follow these when adding to the project)
================================================================================

2.1  Query-kind contract lives in TWO places, keep them in sync:
     Frontend: src/datasource/types.ts       (QueryKind enum + MerakiQuery interface)
     Backend:  pkg/plugin/query/dispatch.go  (QueryKind string constants + MerakiQuery struct)

2.2  Adding a new query kind (7-step recipe):
     1. Add the kind to BOTH src/datasource/types.ts (QueryKind enum) and
        pkg/plugin/query/dispatch.go (constants).
     2. Add an endpoint wrapper in pkg/meraki/ if not already present.
     3. Add an EndpointTimeRange entry in pkg/meraki/timerange.go if the
        endpoint is timeseries.
     4. Create pkg/plugin/query/<kind>.go with a handler:
          func handleXxx(ctx, client, q MerakiQuery, tr TimeRange) (*data.Frame, error)
     5. Wire the handler into dispatch.go's switch.
     6. For metricFind (variable hydration), extend pkg/plugin/query/metricfind.go.
     7. Add to QueryEditor.tsx (src/datasource/QueryEditor.tsx) so users can
        select the kind manually.

2.3  Adding a new scene page (6-step recipe):
     1. src/pages/<Area>/<area>Scene.ts  — EmbeddedScene factory
     2. src/pages/<Area>/<area>Page.ts   — SceneAppPage with routePath
     3. Import the page in src/components/App/App.tsx and add to pages[]
     4. Add an entry to "includes" in src/plugin.json with the URL
     5. Variables: reuse from src/scene-helpers/variables.ts when possible;
        define new factories there, NOT inline in the scene file
     6. Panels: reuse from src/scene-helpers/panels.ts; define new ones there

2.4  Frame column naming (consistency across kinds):
     - Use the Meraki-native JSON field names when they're sensible
       (e.g. "serial", "name", "model", "networkId", "orgId").
     - For historical timeseries, use "ts" for the time column (matches how
       Meraki returns timestamps) and bundle labels on the value field via
       data.Field.Labels.

2.5  DO NOT edit files in .config/ that carry the scaffold warning
     ("DO NOT EDIT THIS FILE DIRECTLY"). EXCEPTIONS already made in v0.1:
     • .config/bundler/copyFiles.ts  — added svg/png globs so nested plugins'
       img/ dirs get copied to dist/<nested>/img/. Note this for future
       scaffold upgrades (may need re-apply).
     • .config/docker-compose-base.yaml  — changed container name + volume
       path + log filter + allow-unsigned to use the new plugin ID.
     • .config/supervisord/supervisord.conf — updated path + binary name.

2.6  Provisioning
     - provisioning/datasources/meraki.yaml: stable UID "rknightion-meraki-ds".
       This UID is hardcoded in src/scene-helpers/datasource.ts as
       MERAKI_DS_UID. If you rename the UID, update both.
     - provisioning/plugins/app.yaml: auto-enables the app plugin on startup.

2.7  Dependencies
     - Frontend: scaffolded with @grafana/create-plugin. No new npm deps
       required for v0.1. New v0.2+ work should prefer @grafana/* first.
     - Backend (go.mod): hashicorp/golang-lru/v2 and golang.org/x/time added
       for v0.1. Further deps should be justified in a PR description.

12. GOTCHAS / THINGS THAT WILL BITE YOU
================================================================================

G.1  Meraki rate limits are PER-ORGANIZATION, not per-key. Exceeding returns
     429 with Retry-After. Our limiter is keyed by orgID for this reason.

G.2  Partial-success 200 responses. Some Meraki endpoints return HTTP 200
     with body `{"items": [...], "errors": [...]}`. client.go detects these
     and surfaces as PartialSuccessError — but the body still contains
     partial data. If a handler wants to present partial data with an error
     notice, it should inspect the error type and extract .items.

G.3  Link header AND startingAfter. client.EachPage/GetAll follows Link
     rel=next. Some newer endpoints use `startingAfter` as a query param
     instead. If a v0.2+ endpoint paginates differently, extend
     pagination.go OR loop in the endpoint wrapper itself.

G.4  Meraki timestamps lag 30-120s behind "now". FreshnessFloor=60s in
     timerange.go subtracts that. Don't remove it or your timeseries tails
     will be empty.

G.5  Timespan caps per endpoint (see KnownEndpointRanges):
     - organizations/.../sensor/readings/history:                 730 days
     - organizations/.../apiRequests(/overview):                    31 days
     - networks/.../wireless/usageHistory:                          31 days
     - organizations/.../wireless/devices/channelUtilization/history: 31 days
     - organizations/.../devices/uplinksLossAndLatency:              5 minutes
     Passing a longer window is a 400. Resolve() clamps.

G.6  Resolution quantization: Meraki only accepts a fixed set per endpoint.
     Always round UP to the nearest allowed bucket. quantizeUp() handles it.

G.7  The nested DS has NO backend. DO NOT add `"backend": true` to
     src/datasource/plugin.json — that would make Grafana try to launch a
     separate gpx_rknightion-meraki-datasource binary that doesn't exist.

G.8  Each plugin instance has its own Go process. If you later add a
     DS-level Go backend, the app and DS will NOT share memory. Design
     around that (e.g. both call the same Meraki client independently, OR
     the DS delegates to the app via resource calls — which is what v0.1
     already does and is the reason the DS is frontend-only).

G.9  The MERAKI_DS_UID constant ("rknightion-meraki-ds") lives in two
     places: provisioning/datasources/meraki.yaml AND
     src/scene-helpers/datasource.ts. Keep them aligned.

G.10 When a user hasn't configured the API key, /resources/query returns 412
     Precondition Failed. Scene queries fail silently today — see polish
     item P.5 above.

G.11 plugin-validator in CI runs against dist/<plugin-id>.zip and currently
     only validates the app plugin (not the nested DS). When we publish,
     we'll need to package both — the release.yml workflow may need
     extension.

G.12 Scaffold tsconfig reports JSX errors in the IDE because it uses a bare
     config when the LSP can't chain through .config/tsconfig.json. These
     are NOT real errors — `npm run typecheck` is authoritative.

G.13 Mage isn't globally installed on dev machines by default. CI installs
     it via magefile/mage-action. Locally: `go install github.com/magefile/mage@latest`.

G.14 .config/ is scaffold-managed. Upgrades via `npx @grafana/create-plugin@latest update`
     may revert our edits in .config/bundler/copyFiles.ts,
     .config/docker-compose-base.yaml, and .config/supervisord/supervisord.conf.
     After any scaffold upgrade, RE-CHECK those files.

G.15 Plugin config save requires `window.location.reload()`, NOT
     `locationService.reload()`. SPA-level reload keeps `plugin.meta` stale in
     the React tree AND does not trigger Grafana to re-instantiate the backend
     plugin with the new secureJsonData. Symptom if you get this wrong: user
     clicks Save, nothing visibly changes, Test Connection stays disabled, and
     other pages report "not configured" until the user manually hits browser
     refresh. See src/components/AppConfig/AppConfig.tsx — it pairs a brief
     success Alert with a ~600ms delayed window.location.reload().

G.16 AppConfig is NOT wrapped in a `<form onSubmit>`. Use a `<div>` with
     `type="button"` + onClick on the Save button. Reasons: (a) Grafana's
     outer plugin-settings page may or may not wrap us in its own <form>
     across versions; a nested submit is fragile, (b) keyboard-enter
     submission on an API-key field is a mild foot-gun (accidental submits).
     The Save button is the only action surface — keep it that way.

G.17 FieldConfig.DisplayNameFromDS is a pre-formatted final string — Grafana
     does NOT template-substitute it. Use FieldConfig.DisplayName with
     ${__field.labels.<name>} if you need interpolation, or bake the final
     string at emit time. We hit this when the CO₂ card legend rendered as
     literal "{{serial}}". See pkg/plugin/query/sensor_readings.go where
     the DisplayNameFromDS is composed per-frame; use the same pattern for
     AP (wireless) and switch legends in v0.2+.

G.18 Native-timeseries frames must be one-frame-per-series with data.Labels
     on the value field. A single long-format frame with
     (ts, serial, metric, value) columns renders as an EMPTY chart — the
     timeseries viz can't infer series grouping from a mixed
     (time, string, number) shape. A client-side partitionByValues transform
     does NOT fix this (the residual string column stays on each partition
     and the viz still ignores it). Backend fix is the correct call.
     See pkg/plugin/query/sensor_readings.go:handleSensorReadingsHistory for
     the canonical shape and pkg/plugin/query/wireless.go:handleWirelessChannelUtil
     for the per-(serial, band) variant.

G.19 Query-handler signature takes Options as the 5th argument (LabelMode
     etc.). The dispatcher threads plugin settings through Handle → runOne →
     handler. New handlers MUST accept the param even when unused (use
     `_ Options`). See pkg/plugin/query/dispatch.go — `Options` struct +
     `handlerFn` type.

G.20 Prefer server-side KPI aggregation over client-side filterByValue +
     reduce transform chains. Client transforms silently fall back to the
     wrong reducer in some Grafana versions; the sensorAlertSummary kind
     (pkg/plugin/query/sensor_summary.go) exists because of this. Every
     future KPI row (alerts counts via alertsOverview, switch port totals,
     AP availability) should follow the same "one wide frame, one field per
     KPI" pattern.

--------------------------------------------------------------------------------
16.1  MERAKI-DOCUMENTED LIMITS (authoritative numbers)
--------------------------------------------------------------------------------

Organization (our primary constraint; rate limiter keys on orgID):
  - Steady-state: 10 requests per second per organization.
  - Burst:        10 requests in the first second; max 30 requests in 2s.
  - Shared across ALL API applications using the organization's API key.
  - Exceeding -> HTTP 429 + Retry-After header. Meraki's own guidance: "Pause
    for one or two seconds before making the next API call" and "increase wait
    times during high-volume periods".

Source IP (secondary; effectively unreachable from one Grafana instance unless
you fan out to many orgs):
  - 100 requests per second per source IP, shared across all clients using
    that address.
  - Relevant if we ever run this plugin with many orgs off one Grafana
    instance, OR if multiple tenants share an egress IP.

Current code posture (as of v0.2.0):
  - pkg/meraki/ratelimit.go: token bucket, 10 rps, burst 20, ±10% jitter,
    keyed on orgID. MATCHES the per-org cap.
  - We do NOT model the IP-level 100 rps ceiling. At one instance x 10-org
    concurrency we'd hit 100 rps total — still within the cap — but it would
    be wise to add an OPT-IN global bucket for safety. See §16.4-G below.

--------------------------------------------------------------------------------
16.2  USER-AGENT COMPLIANCE   (P.9 is the immediate-next action item)
--------------------------------------------------------------------------------

Required format per Meraki docs:
  "ApplicationName VendorName"
or with version:
  "ApplicationName/1.0 VendorName"

Rules (all required, not optional):
  - Omit all spaces, hyphens, and special characters from ApplicationName AND
    VendorName. So "Grafana-Meraki-App" is invalid (hyphens); so is
    "Grafana Meraki" (space in the app name).
  - Forward slashes are reserved for version info only.
  - Must be consistent/correct across every request from the application.
  - "Do not omit the user agent string from any of your requests."

Why it matters beyond spec compliance: Meraki states the UA "helps your
customers and Meraki administrators monitor API interactions with their
environments and helps build their trust." Customers can see per-app traffic
attribution in their org's API Analytics dashboard. A compliant, versioned,
branded UA also makes this plugin findable in Cisco's ecosystem reports.

Current value: "Grafana-Meraki-App" (pkg/meraki/client.go:22,
pkg/plugin/app.go:128). NON-COMPLIANT — hyphens.

Target value: "GrafanaMerakiPlugin/<semver> rknightion"
  - Semver pulled from a new pkg/meraki/version.go (const ClientVersion),
    bumped each release in lockstep with CHANGELOG.md + package.json.
  - Vendor "rknightion" matches current plugin-ID namespace; revisit when
    §Q.7 rename to robknight-* happens (vendor token moves to "robknight").

Tracked as polish item P.9 (§4). Do this before the next signed release.

16.3  MERAKI-RECOMMENDED API-EFFICIENCY LEVERS
--------------------------------------------------------------------------------

Meraki's rate-limit page lists four techniques for reducing request volume.
Status + proposed action in our plugin:

  A. "Use organization-wide API endpoints instead of per-device operations."
     Example: `getOrganizationDevices` over `getDevice`.
     Status: ALREADY FOLLOWED. pkg/meraki/devices.go uses
     GetOrganizationDevices(orgID); pkg/meraki/switches.go uses
     switch/ports/statuses/bySwitch; sensors uses the org-level readings
     endpoint. Document this as a hard rule in §2 (Code conventions) so
     future endpoint wrappers don't regress.
     Action: add a bullet to §2.2 / §2.7 on next doc pass — "Always prefer
     the org-wide endpoint; per-device endpoints are last resort."

  B. "Use Action Batches to bundle configuration requests."
     Status: NOT APPLICABLE TO v0.x. The plugin is read-only today (all query
     handlers call GET; resources.go exposes /ping, /query, /metricFind but
     never POST/PUT/DELETE to Meraki). Action Batches are POST-only and
     only useful when the plugin gains write capabilities.
     Action: park. Revisit if v1+ adds "Configure from Grafana" flows (e.g.
     an alert-rule auto-provisioner writing to Meraki webhook settings).

  C. "Use getOrganizationConfigurationChanges instead of frequent polling."
     Endpoint: GET /organizations/{orgId}/configurationChanges
     Params: t0, t1, timespan. Cursor pagination (startingAfter/endingBefore).
     Use case: a "Change Log" scene showing who changed what, when, across
     the org. Replaces scenarios where a user today might poll devices or
     networks to detect drift.
     Proposal: add NEW query kind `ConfigurationChanges` in a future phase
     (suggested v0.6, alongside licensing/api-usage; see §9).
     TTL: 5m (audit log is cold; we only need near-freshness for the most
     recent entries).
     New files: pkg/meraki/configuration_changes.go (endpoint wrapper) +
     pkg/plugin/query/configuration_changes.go (handler) + a scene under
     src/pages/AuditLog/.

  D. "Replace repetitive polling with change-history endpoints like
      getOrganizationDevicesAvailabilitiesChangeHistory."
     Endpoint: GET /organizations/{orgId}/devices/availabilities/changeHistory
     Params: t0, t1, timespan, perPage (3-1000), startingAfter, endingBefore,
     networkIds, serials, productTypes.
     Use case: instead of re-reading the full availability snapshot (our
     current DeviceAvailabilities query kind, TTL 1m), a user on a long
     dashboard refresh interval can ask for only the *changes* since the
     last render.
     Proposal: add NEW query kind `DeviceAvailabilityChanges` in a future
     phase (v0.3+ or bundled with v0.7 MG/MV phase).
     TTL: 30s (small deltas, cheap to re-pull).
     DO NOT replace the existing DeviceAvailabilities kind — change-history
     is additive. Operators with short refresh intervals still want the
     snapshot; operators building "device flaps over 24h" panels want the
     changeHistory feed.
     New files: add GetOrganizationDevicesAvailabilitiesChangeHistory() to
     pkg/meraki/devices.go (supports Link header AND startingAfter cursor —
     extend pagination.go if change-history endpoints prefer the cursor
     variant; verify against a live org before coding).

  E. "Monitor actual usage via API Analytics in the Dashboard."
     Status: user-facing guidance, not a plugin change.

16.4  CACHING + REQUEST-HYGIENE IMPROVEMENTS
--------------------------------------------------------------------------------

Current state (pkg/meraki/cache.go):
  - TTLCache: golang-lru/v2 + per-entry expiry.
  - 2048-entry LRU shared across every endpoint, every org.
  - Per-kind TTLs enumerated in §1.7.
  - Keyed on sha256(orgID + path + sorted params).
  - Scope: per plugin-instance, in-memory. No Redis, no cross-replica share.

Improvements to consider, in priority order:

  A. Singleflight / request coalescing.
     Problem: when a user opens a dashboard with 5 panels all querying
     getOrganizationDevices for the same org, we fire 5 concurrent HTTP
     requests because the cache is checked per-request. Only AFTER the first
     one lands does the TTL prevent duplicate requests.
     Fix: wrap Client.Get / Client.GetAll in golang.org/x/sync/singleflight.
     Key = the existing CacheKey(orgID, path, params). Result: N concurrent
     lookups for the same key share ONE in-flight request.
     Estimated reduction: the worst offender (Home page, 6+ panels on
     initial load, all hitting Organizations / Devices / Alerts) goes from
     ~15 requests to ~3-4 on a cold cache. On a warm cache: no change.
     Files: pkg/meraki/client.go (wrap Get / GetAll + add a per-Client
     singleflight.Group), pkg/meraki/client_test.go (table test asserting N
     parallel Get calls land in 1 HTTP round-trip when keys match).
     Dep add: golang.org/x/sync (already transitively in go.mod? verify
     before adding).
     Priority: HIGH — single biggest win for typical dashboard load.

  B. Cache-key jitter on TTL (avoid thundering-herd at expiry).
     Problem: every Organizations cache entry set at roughly the same time
     expires at the same time, so the next dashboard refresh after 1h
     stampedes ALL handlers in parallel.
     Fix: when setting a TTL, randomise within ±10% of the configured value.
     Pairs naturally with the rate-limiter jitter (§1.6).
     Files: pkg/meraki/cache.go:Set (wrap ttl with applyJitter before
     computing expires).
     Priority: MEDIUM — mitigates a specific failure mode but singleflight
     (16.4-A) addresses the same root cause with bigger payoff.

  C. Stale-while-revalidate (SWR).
     Problem: when a TTL expires mid-dashboard-refresh, the user sees a
     spinner while we re-fetch. For high-cardinality slow endpoints
     (sensor history, channelUtil history), this stings.
     Fix: expose a "stale_grace" window alongside TTL. While t is between
     (expires, expires+stale_grace), serve the cached value immediately AND
     fire an async refresh. After stale_grace, fall back to a synchronous
     fetch.
     Files: pkg/meraki/cache.go (new Entry.StaleUntil + Cache.GetStale
     method returning (value, stale bool, ok bool)); client.go (refactor
     Get / GetAll to take a staleGrace param; dispatch refresh in a goroutine
     gated by singleflight from 16.4-A so we don't stampede the refresh).
     Per-kind stale_grace proposals:
       Organizations:                1h +  30m stale
       Networks:                    15m +  10m stale
       Devices / DeviceStatusOverv:  5m +   2m stale
       SensorReadingsHistory:        1m +  30s stale
       Alerts:                      30s +  15s stale
     Priority: MEDIUM-HIGH — dashboard "snappiness" improvement without
     extra API load if combined with singleflight.

  D. Negative caching of 404 / 412.
     Problem: if a query refers to a deleted network, every re-render pays
     a full API round-trip for the 404.
     Fix: cache negative responses with a short TTL (30s-60s).
     Scope: only 404 for resource-not-found shapes. Do NOT cache 401/403
     (user may rotate the key) or 5xx (transient). 412 from our own
     /query / /metricFind (API key unset) is already handled by
     ConfigGuard on the frontend — skip.
     Files: pkg/meraki/client.go:Do (on terminal non-2xx, write a negative
     entry in cache tagged as such), cache.go (add tri-state getter).
     Priority: LOW — edge case. Only worth it after telemetry shows repeat
     404s.

  E. Cache size tuning + per-org eviction fairness.
     Current: a single 2048-entry LRU for the whole plugin instance.
     Concern: a big-org operator with 50 networks + 500 devices evicts a
     small-org operator's entries before their TTL fires.
     Fix: partition the cache PER ORG (map[string]*TTLCache), each capped at
     e.g. 512 entries. OR keep one LRU but segment by orgID prefix and
     evict from the largest segment first.
     Files: pkg/meraki/cache.go (new partitioned type) OR app.go (wire a
     map instead of a single cache).
     Priority: LOW — revisit only if operators report multi-org fairness
     issues.

  F. Optional external cache for multi-replica deployments.
     See §Q.3 (currently "not planned for v0.x"). Same decision stands —
     rely on 429 + Retry-After across replicas, plus the sharedFraction
     knob. Revisit only if operators report cold-cache stampedes at scale.

  G. Opt-in IP-level rate limiter.
     See §16.1 above. Only relevant when multiple orgs fan out from one
     instance (aggregate > 100 rps on a single egress IP).
     Fix: add a SECOND token bucket keyed on the literal "ip" with
     RequestsPerSecond=100, Burst=200, SharedFraction=sharedFraction. Both
     the per-org and the per-ip limiter must grant before Do() fires the
     request.
     Files: pkg/meraki/ratelimit.go (accept multiple bucket keys per
     Acquire) OR layer a second RateLimiter in client.go.
     Priority: LOW-DEFER — only useful for multi-tenant / many-org
     deployments.

16.5  GRAFANA-SIDE CACHING INTERACTIONS
--------------------------------------------------------------------------------

Grafana itself provides query result caching (Enterprise) and backend plugin
caching primitives. Worth noting BEFORE we reach for them:

  - Grafana's backend plugin cache (data.Cache in the SDK) caches the final
    data.Frames, not raw responses. It's complementary to our TTLCache, not
    a replacement — we cache BEFORE frame assembly so a single cache entry
    can feed different frame shapes (e.g. stat vs timeseries). No change
    proposed.
  - Grafana query result caching (Enterprise) sits in FRONT of the plugin.
    When enabled, the plugin's Query() isn't called until the Grafana-level
    TTL expires. Our TTLs remain correct but effective TTL becomes
    max(our_ttl, grafana_ttl). Document this in README once §16.4
    improvements ship so operators know not to double-tune TTLs.
  - "Query result reuse via instance manager" — NewApp() is called per
    plugin instance; Dispose() on rebuild. Our cache lifetime matches
    instance lifetime. Grafana may rebuild instances on setting changes,
    flushing our cache — that's EXPECTED and desirable (settings changes
    can invalidate key hashes).

Rate-limit page (https://developer.cisco.com/meraki/api-v1/rate-limit/):
  - "10 requests per second per organization."
  - "10 requests in the first second (max 30 requests in 2s)."
  - "100 requests per second" per source IP, "shared among all clients
     using that address."
  - Org cap is "shared across all API applications using the organization's
     API key."
  - 429 response: "The response includes a Retry-After header indicating
     wait duration before the next request."
  - Recommended handling: "Pause for one or two seconds before making the
     next API call" and "increase wait times during high-volume periods."
  - Action Batches "bundle multiple configuration requests into a single
     transaction" — "bulk POST, PUT, and DELETE operations."
  - "Use organization-wide API endpoints instead of per-device operations"
     (example given: getOrganizationDevices vs getDevice).
  - "Implement getOrganizationConfigurationChanges to retrieve snapshots
     instead of polling frequently."
  - "Replace repetitive polling with change-history endpoints like
     getOrganizationDevicesAvailabilitiesChangeHistory."
  - "Monitor actual usage via API Analytics in the dashboard."

User-Agent page (https://developer.cisco.com/meraki/api-v1/user-agents-overview/):
  - Required format: "ApplicationName VendorName" — optional with version
    "ApplicationName/1.0 VendorName".
  - Application and Vendor names must "omit all spaces, hyphens, and
    special characters".
  - "Use only forward slashes (/) for version information."
  - Must be "consistent or correct...across your application."
  - "Provide the user agent string in the user agent header of every API
    request" and "Do not omit the user agent string from any of your
    requests."
  - Benefit: "helps your customers and Meraki administrators monitor API
    interactions with their environments and helps build their trust."

Action Batches overview
(https://developer.cisco.com/meraki/api-v1/action-batches-overview):
  - POST / GET / PUT / DELETE on /organizations/{orgId}/actionBatches.
  - `confirmed` + `synchronous` flags; nested `actions[]` with
    `operation` (create/update/delete) + `resource` + `body`.
  - Only relevant once the plugin grows write-side features. Parked.
