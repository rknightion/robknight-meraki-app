    4. Navigate any of the 12 pages:

       - Organizations: clickable inventory -> tabbed detail (Overview /
         Devices / Alerts).
       - Appliances (MX): inventory + uplink status + uplinks-overview KPIs +
         per-MX drilldown (Overview / Uplinks / VPN / Firewall). Uplinks tab
         shows native loss/latency timeseries; VPN tab shows a peer matrix;
         Firewall tab shows port-forwarding + appliance settings.
       - Access Points (MR): channel-util + SSID usage timeseries + clickable
         AP inventory; per-AP drilldown (Overview / Clients / RF).
       - Switches (MS): inventory + per-switch drilldown (Overview / Ports)
         with a speed-coloured port map and per-port packet counters.
       - Cameras (MV): inventory + onboarding status; per-camera drilldown
         (Overview / Analytics / Zones).
       - Cellular Gateways (MG): inventory + fleet signal chart; per-gateway
         drilldown (Overview / Uplink with RSRP/RSRQ gauges / Port
         Forwarding).
       - Sensors (MT): native timeseries per metric + per-sensor detail
         page with auto-hide-on-empty behaviours.
       - Insights: tabbed page (Licensing / API Usage / Clients) surfacing
         licenses, response-code interval chart, top-N usage tables.
       - Events: network events with product-type filter, hourly timeline
         bar chart, events table with cross-family drilldown.
       - Alerts: severity filter + hourly timeline + sortable alerts table
         with per-row cross-family drilldown (v0.4.0).
       - Audit Log: org-scoped configuration-change feed with timeline + table,
         filterable by network and admin ID (v0.4.0).
    5. Every scene shows a "not configured" banner (linking to the in-app
       Configuration page) when the API key is missing.

• Cache + client stack hardening (v0.4.0, §7 completion):
    - User-Agent:  "GrafanaMerakiPlugin/<version> rknightion" (spec-compliant).
    - Singleflight: N concurrent Get/GetAll calls for the same cache key
      collapse to 1 HTTP round-trip.
    - Cache:        per-org partitioned LRU + TTL jitter (±10%) +
      stale-while-revalidate (TTL/2 capped at 30m) + negative-404 caching

      per-org bucket (for multi-tenant deployments).

1. LOCKED-IN ARCHITECTURE DECISIONS (do not revisit without strong reason)
================================================================================

1.1  Plugin ID namespace
     - Current IDs: "rknightion-meraki-app"        (app plugin)
                    "rknightion-meraki-datasource" (nested data source)
     - Go module: github.com/robknight/grafana-meraki-plugin
     - At first SIGNED release, plan to migrate to "robknight-meraki-app" +
       "robknight-meraki-datasource". Tracked in §4 as a pending task (Q.7).
     - The Go module path can stay as-is. Plugin ID and Go module path are
       independent.

1.2  Plugin type: App + nested DS + shared Go backend
     - The APP plugin has a Go backend (gpx_meraki). The app owns:
       • the Meraki API key (in app.secureJsonData.merakiApiKey)
       • the meraki.Client instance (rate limiter + cache + HTTP client)
       • resource endpoints: POST /resources/query, POST /resources/metricFind,
         GET /resources/ping
       • CheckHealth (calls GET /organizations)
     - The DATA SOURCE plugin is FRONTEND-ONLY. It has no backend binary.
       Its datasource.ts implements query() and metricFindQuery() by POSTing to
       the app's resource endpoints via getBackendSrv().fetch.
     - Rationale: one place to configure the API key, one Go binary to build,
       one rate limiter / cache shared across every query and every dashboard
       panel. Same pattern as Grafana's Synthetic Monitoring app.

1.3  NO Prometheus / exporter dependency
     - The plugin does NOT depend on a Prometheus data source.
     - The plugin does NOT scrape, read, or import anything from the reference
       Python exporter at /Users/rob/repos/meraki-dashboard-exporter.
     - The exporter is READ-ONLY REFERENCE MATERIAL for:
       • Which Meraki endpoints exist and what they return
       • Per-endpoint rate-limit + pagination quirks
       • Dashboard panel layouts to reproduce as scenes
     - Every query from a scene goes directly to api.meraki.com via our Go
       backend — "Infinity-style" but with a typed query-kind contract.

1.4  Scenes everywhere. No provisioned static JSON dashboards.
     - Every observability flow is built as a @grafana/scenes page under
       src/pages/<Area>/. DO NOT add JSON dashboards under
       provisioning/dashboards/.

1.5  Authentication
     - API key stored on the APP plugin in secureJsonData.merakiApiKey. Read
       server-side only via settings.DecryptedSecureJSONData. NEVER expose the
       key to the browser.
     - The nested DS has NO API-key field (ConfigEditor shows a message
       directing users to the app config page).

1.6  Rate limiting
     - Per-instance, per-org token bucket. 10 req/s default, burst 20, jitter
       ±10%. Implementation: pkg/meraki/ratelimit.go (token bucket with test
       hooks for Clock/Sleep/Rand).
     - Config knob: AppJsonData.sharedFraction (0 < x <= 1). Operators with N
       Grafana replicas set this to 1/N.
     - Distributed limiter is INTENTIONALLY deferred. Rely on the server-side
       429 + Retry-After as the natural coordinator across replicas.

1.7  Caching
     - In-memory TTL LRU, pkg/meraki/cache.go (wraps hashicorp/golang-lru/v2).
     - Key: sha256(orgID + path + sorted params) via CacheKey().
     - Per-kind TTLs (keep consistent with what's used in pkg/plugin/query/*):
         Organizations:                        1h
         Networks:                             15m
         Devices:                              5m
         DeviceStatusOverview:                 1m
         SensorReadingsLatest / Summary:       30s
         SensorReadingsHistory:                1m
         Wireless usage / channel util:        1m
         NetworkSsids:                         5m
         ApClients:                            1m
         DeviceAvailabilities:                 1m
         Alerts / AlertsOverview:              30s
         Appliance uplink statuses / overview: 30s
         Appliance VPN statuses:               30s
         Appliance VPN stats:                  1m
         Device uplinks loss/latency:          30s
         Appliance port forwarding / settings: 5m
         Licenses overview / list:             15m
         ApiRequests overview:                 1m
         ApiRequests byInterval:               5m
         ClientsOverview + top-N:              5m
         Camera onboarding:                    5m
         Camera analytics overview / history:  1m
         Camera analytics live:                30s
         Camera analytics zones:               5m
         Camera retention profiles:            15m
         MG uplinks:                           1m
         MG port forwarding / LAN / connmon:   15m
         Network events:                       30s
     - No Redis, no cross-replica cache. Per-plugin-instance only.

1.8  Time-range handling
     - pkg/meraki/timerange.go defines EndpointTimeRange with MaxTimespan and
       AllowedResolutions, plus KnownEndpointRanges map.
     - Resolve() applies:
         • 60s freshness floor (subtract from `now` before computing t1)
         • Timespan clamp to MaxTimespan (emits Annotation if truncated)
         • Resolution quantization UPWARD to nearest allowed bucket
     - KnownEndpointRanges currently covers: sensor history (730d),
       apiRequests (31d), apiRequests byInterval (31d), wireless usage
       history (31d), channel util history (31d), uplinksLossAndLatency
       (5min), appliance VPN stats (31d), clients overview (31d),
       summary/top/* (186d), camera analytics overview (7d).
     - ADD new entries whenever a new timeseries endpoint lands.

1.9  Snapshot vs. timeseries conventions
     - Native timeseries only from: sensor readings history, wireless usage
       history, channel utilization history, uplinksLossAndLatency, camera
       analytics overview + zone history, apiRequests byInterval.
     - Snapshot endpoints (deviceStatusOverview, deviceStatuses, switch port
       statuses, VPN statuses, appliance uplinks, MG uplinks, camera live)
       return point-in-time frames meant for Stat or Table viz. DO NOT try to
       synthesize timeseries from snapshots.
     - Future: a tiny opt-in 24h x 1min in-process ring buffer for
       deviceStatuses state transitions — NOT implemented. Tracked in §4 as a
       deferred feature.

1.10 Frame-notice error pattern
     - Per-query errors become `data.Notice{Severity: Error, Text: ...}` on a
       placeholder frame. A single bad query does NOT kill the whole panel.
     - See pkg/plugin/query/dispatch.go for the wrapping logic. Keep this
       pattern for every new query kind.

1.11 URL-sync for scenes
     - Every scene page with drilldowns or tabs uses `routePath: 'path/*'`
       with a trailing `*`. Deep links must be bookmarkable.

1.12 Cross-family drilldown via server-computed drilldownUrl column
     - Handlers that emit device rows (devices, deviceAvailabilities,
       cameraOnboarding, mgUplinks, topDevices, topSwitchesByEnergy,
       networkEvents) include a `drilldownUrl` column computed by
       pkg/plugin/query/device_urls.go::deviceDrilldownURL(prefix, productType,
       serial).
     - The dispatcher threads the `/a/<plugin-id>` prefix through
       Options.PluginPathPrefix (populated in pkg/plugin/resources.go).
     - Frontend tables consume the column via `${__data.fields.drilldownUrl}`
       in override links, then hide the column via an `organize` transform.
     - Grafana's URL template engine cannot conditionally branch on a row's
       productType, so a backend-computed column is the only clean path to
       heterogeneous device links in one table. Do NOT reintroduce
       hardcoded per-family URLs in frontend tables that span product
       families.

2. CODE CONVENTIONS (follow these when adding to the project)
================================================================================

2.1  Query-kind contract lives in TWO places, keep them in sync:
     Frontend: src/datasource/types.ts       (QueryKind enum + MerakiQuery interface)
     Backend:  pkg/plugin/query/dispatch.go  (QueryKind string constants + MerakiQuery struct)

2.2  Adding a new query kind (7-step recipe):
     1. Add the kind to BOTH src/datasource/types.ts (QueryKind enum) and
        pkg/plugin/query/dispatch.go (constants).
     2. Add an endpoint wrapper in pkg/meraki/ if not already present. ALWAYS
        prefer org-wide endpoints over per-device ones (see §7.3-A for
        rationale — Meraki documentation calls this out explicitly).
     3. Add an EndpointTimeRange entry in pkg/meraki/timerange.go if the
        endpoint is timeseries.
     4. Create pkg/plugin/query/<kind>.go with a handler:
          func handleXxx(ctx, client, q, tr, opts) ([]*data.Frame, error)
     5. Wire the handler into dispatch.go's handlers map.
     6. For metricFind (variable hydration), extend pkg/plugin/query/metricfind.go.
     7. Add to QueryEditor.tsx (src/datasource/QueryEditor.tsx) so users can
        select the kind manually.

2.3  Adding a new scene page (6-step recipe):
     1. src/pages/<Area>/<area>Scene.ts  — EmbeddedScene factory
     2. src/pages/<Area>/<area>Page.ts   — SceneAppPage with routePath
     3. Import the page in src/components/App/App.tsx and add to pages[]
     4. Add an entry to "includes" in src/plugin.json with the URL
     5. Variables: reuse from src/scene-helpers/variables.ts when possible.
        `deviceVariable({name, label, productType})` and
        `networkVariableForProductTypes([…])` already exist as generic
        factories — use them instead of copying per-area boilerplate.
     6. Panels: area-local panels.ts files own their own `oneQuery` + panel
        factories (precedent set by AccessPoints/Switches). Reusable KPI
        patterns go in src/scene-helpers/panels.ts.

2.4  Frame column naming (consistency across kinds):
     - Use the Meraki-native JSON field names when they're sensible
       (e.g. "serial", "name", "model", "networkId", "orgId").
     - For historical timeseries, use "ts" for the time column (matches how
       Meraki returns timestamps) and bundle labels on the value field via
       data.Field.Labels.
     - For device-bearing frames that could route to any product family, emit
       a `drilldownUrl` column using deviceDrilldownURL() (§1.12).

2.5  DO NOT edit files in .config/ that carry the scaffold warning
     ("DO NOT EDIT THIS FILE DIRECTLY"). EXCEPTIONS already made:
     • .config/bundler/copyFiles.ts  — added svg/png globs so nested plugins'
       img/ dirs get copied to dist/<nested>/img/. Note this for future
       scaffold upgrades (may need re-apply).
     • .config/docker-compose-base.yaml  — changed container name + volume
       path + log filter + allow-unsigned to use the new plugin ID.
     • .config/supervisord/supervisord.conf — updated path + binary name.

2.6  Provisioning
     - provisioning/datasources/meraki.yaml: stable UID "rknightion-meraki-ds".
       This UID is hardcoded in src/scene-helpers/datasource.ts as
       MERAKI_DS_UID. If you rename the UID, update both.
     - provisioning/plugins/app.yaml: auto-enables the app plugin on startup.

2.7  Dependencies
     - Frontend: scaffolded with @grafana/create-plugin. Prefer @grafana/*
       first. React 18, @grafana/scenes ^7, @grafana/ui/data/runtime 12.4.2.
     - Backend (go.mod): hashicorp/golang-lru/v2 added for v0.1; v0.2.0 `go
       mod tidy` dropped golang.org/x/time. Further deps should be justified
       in a PR description.

4.5  V0.6 ROADMAP — BUNDLED ALERT RULES + INSTALL/UNINSTALL INFRASTRUCTURE
     [PLANNED — brainstormed 2026-04-18; supersedes §4.2 Q.6]
--------------------------------------------------------------------------------

Goal: ship 13 Grafana-managed alert rule templates with an in-plugin
install/uninstall UX. The plugin calls Grafana's alert-provisioning API
via a plugin service account (Grafana 10.3+ IAM, already satisfied by
our grafanaDependency >=12.3.0).

Scope EXCLUDES: see §4.4 exclusions plus the 5 alert templates dropped
during brainstorming (MG signal-trend, camera retention, firmware-pending,
API 429, config-change-rate).

4.5.1  LOCKED-IN ARCHITECTURE DECISIONS FOR V0.6
  a. Plugin calls Grafana's own provisioning API. Authenticated via the
     plugin service-account token that Grafana injects automatically
     when the `iam` section is present in plugin.json (Grafana 10.3+).
     NO user-provided service-account token; NO YAML download/drop path.
  b. Install UX: grouped toggles (per-group install/uninstall). Each
     group contains 1–3 rule templates. 7 groups total:
       availability, wan, wireless, cellular, sensors, cameras, lifecycle.
  c. Threshold model: configurable in plugin UI. AppJsonData stores user
     overrides; baked defaults live in the rule templates.
  d. Org scope: auto-fan-out to every Meraki org visible via the API key.
     Reconciler fetches orgs via the existing meraki client at reconcile
     time — no additional storage of org lists.
  e. Rule UID scheme: `meraki-<group>-<rule>-<orgId>`. Prefix `meraki-`
     is STABLE across plugin-ID rename (Q.7). Plugin-ID is NEVER in the
     UID.
  f. Rule folder: "Meraki (bundled)". Folder UID `meraki-bundled-folder`.
     Reconciler creates the folder if missing.
  g. Deletion safety: reconciler only deletes rules matching BOTH the
     UID prefix AND the label `managed_by=meraki-plugin`. Protects users
     who hand-authored similarly-named rules.
  h. Plugin config UI is source of truth for thresholds. User-edits
     inside Grafana's Alerting UI are reverted on next reconcile.
     Documented in the config-UI section and CHANGELOG.
  i. Contact points / notification policies are the user's concern.
     Rules emit standard labels (§4.5.10); users route via those.
  j. Feature-toggle check: CheckHealth probes for externalServiceAccounts
     availability at startup; if unavailable, Home shows a banner
     directing the user to enable the toggle OR upgrade Grafana.

4.5.4  PHASE 2 — RECONCILER
  File: pkg/plugin/alerts/reconciler.go
  Signature:
    func Reconcile(ctx, grafanaClient, merakiClient, desired DesiredState)
                  → ReconcileResult, error
  DesiredState:
    {
      groups: map[groupId]GroupState
      thresholds: map[groupId]map[templateId]map[thresholdKey]any
      orgOverride: []string    // OPTIONAL — empty = "all orgs from merakiClient"
    }
  Algorithm (idempotent, pure function over inputs):
    1. orgs := desired.OrgOverride OR merakiClient.GetOrganizations()
    2. desiredRules := []AlertRule{}
       for each enabled (group, template) × org:
         render(template, threshold_overrides, orgId) → rule
         rule.UID = fmt.Sprintf("meraki-%s-%s-%s", groupId, templateId, orgId)
         desiredRules = append(desiredRules, rule)
    3. currentRules := grafanaClient.ListAlertRules(folderUID)
       filter to those with UID prefix "meraki-" AND label
       managed_by=meraki-plugin.
    4. diff:
         create = desired ∖ current (by UID)
         update = intersect where content differs
         delete = current ∖ desired
    5. Apply sequentially (per-call fail-tolerant, logged, aggregated):
         folder := grafanaClient.EnsureFolder("meraki-bundled-folder",
                                              "Meraki (bundled)")
         for create:  POST  /api/v1/provisioning/alert-rules
         for update:  PUT   /api/v1/provisioning/alert-rules/{uid}
         for delete:  DELETE /api/v1/provisioning/alert-rules/{uid}
    6. Return ReconcileResult{Created, Updated, Deleted, Failed}.

  Unit tests with a httptest.NewServer emulating the Grafana provisioning
  API. Cases:
    - Empty → 9 rules installed (availability+wan+sensors default thresholds).
    - Installed → threshold change → update-only flow.
    - Installed → group disabled → delete-only flow.
    - Installed → new Meraki org → fan-out create.
    - Installed → removed Meraki org → delete stragglers.
    - Plugin rename (datasourceUid changes) → update-all flow.
    - Grafana API returns 500 on one rule → other rules still apply;
      failure reported in Failed slice.

4.5.11 RULE UID SCHEME
  Format:  meraki-<groupId>-<templateId>-<orgId>
  Examples:
    meraki-availability-device-offline-987654
    meraki-wan-uplink-down-987654
    meraki-sensors-readings-out-of-range-987654
  Invariants:
    - No plugin-ID substring. Survives Q.7 rename.
    - Deterministic: same inputs → same UID, enabling idempotent apply.
    - Meraki org IDs are stable; no re-minting when user rotates API keys
      (unless new key grants access to new orgs → new UIDs for those).

4.5.12 PLUGIN-RENAME (Q.7) INTERACTION
  When rknightion-* → robknight-* rename ships:
    1. Every rule's query.datasourceUid changes (was rknightion-meraki-ds,
       becomes robknight-meraki-ds — assuming MERAKI_DS_UID also flips).
       OR decision: KEEP MERAKI_DS_UID on the old string to skip this
       migration. Document whichever path is chosen at rename time.
    2. Every rule's labels unchanged.
    3. UIDs unchanged.
    4. Reconciler re-run on next plugin start; diff surfaces every rule
       as "update" because datasourceUid changed. Result: ~N rules
       silently migrated on first post-rename restart.
  Operational guidance: include "run one reconcile after deploy" as a
  bullet in the rename atomic-commit's CHANGELOG entry.

       surface a warning banner on the config UI if the /status call

================================================================================
5. GOTCHAS / THINGS THAT WILL BITE YOU
================================================================================

G.1  Meraki rate limits are PER-ORGANIZATION, not per-key. Exceeding returns
     429 with Retry-After. Our limiter is keyed by orgID for this reason.

G.2  Partial-success 200 responses. Some Meraki endpoints return HTTP 200
     with body `{"items": [...], "errors": [...]}`. client.go detects these
     and surfaces as PartialSuccessError — but the body still contains
     partial data. If a handler wants to present partial data with an error
     notice, it should inspect the error type and extract .items.

G.3  Link header AND startingAfter. client.GetAll follows Link rel=next. Some
     newer endpoints use `startingAfter` as a query param instead. The
     NetworkEvents wrapper (pkg/meraki/events.go) is the current reference
     for cursor-style pagination handled in the wrapper itself. If a new
     endpoint paginates differently, extend pagination.go OR loop in the
     endpoint wrapper.

G.4  Meraki timestamps lag 30-120s behind "now". FreshnessFloor=60s in
     timerange.go subtracts that. Don't remove it or your timeseries tails
     will be empty.

G.5  Timespan caps per endpoint (see KnownEndpointRanges for the full table):
     - sensor/readings/history:                      730 days
     - apiRequests(/overview):                       31 days
     - apiRequests byInterval:                       31 days
     - wireless usageHistory:                        31 days
     - wireless channelUtilization/history:          31 days
     - devices/uplinksLossAndLatency:                5 minutes
     - appliance/vpn/stats:                          31 days
     - clients/overview:                             31 days
     - summary/top/*:                                186 days (25m floor for switches-by-energy)
     - camera/analytics/overview:                    7 days
     Passing a longer window is a 400. Resolve() clamps and emits a truncation
     annotation.

G.6  Resolution quantization: Meraki only accepts a fixed set per endpoint.
     Always round UP to the nearest allowed bucket. quantizeUp() handles it.

G.7  The nested DS has NO backend. DO NOT add `"backend": true` to
     src/datasource/plugin.json — Grafana would try to launch a separate
     gpx_rknightion-meraki-datasource binary that doesn't exist.

G.8  Each plugin instance has its own Go process. If a future design adds a
     DS-level Go backend, the app and DS will NOT share memory. Design
     around that (e.g. both call the same Meraki client independently, OR
     the DS delegates to the app via resource calls — which is what we do
     today, and the reason the DS is frontend-only).

G.9  The MERAKI_DS_UID constant ("rknightion-meraki-ds") lives in two
     places: provisioning/datasources/meraki.yaml AND
     src/scene-helpers/datasource.ts. Keep them aligned.

G.10 When a user hasn't configured the API key, /resources/query returns 412
     Precondition Failed. Scene queries render nothing; the ConfigGuard
     banner (P.5) informs the user.

G.11 plugin-validator in CI runs against dist/<plugin-id>.zip and currently
     only validates the app plugin (not the nested DS). When we publish,
     we'll need to package both — the release.yml workflow may need
     extension.

G.12 Scaffold tsconfig reports JSX errors in the IDE because it uses a bare
     config when the LSP can't chain through .config/tsconfig.json. These
     are NOT real errors — `npm run typecheck` is authoritative.

G.13 Mage isn't globally installed on dev machines by default. CI installs
     it via magefile/mage-action. Locally: `go install github.com/magefile/mage@latest`.

G.14 .config/ is scaffold-managed. Upgrades via `npx @grafana/create-plugin@latest update`
     may revert our edits in .config/bundler/copyFiles.ts,
     .config/docker-compose-base.yaml, and .config/supervisord/supervisord.conf.
     After any scaffold upgrade, RE-CHECK those files.

G.15 Plugin config save requires `window.location.reload()`, NOT
     `locationService.reload()`. SPA-level reload keeps `plugin.meta` stale in
     the React tree AND does not trigger Grafana to re-instantiate the backend
     plugin with the new secureJsonData. Symptom if you get this wrong: user
     clicks Save, nothing visibly changes, Test Connection stays disabled, and
     other pages report "not configured" until the user manually hits browser
     refresh. See src/components/AppConfig/AppConfig.tsx — it pairs a brief
     success Alert with a ~600ms delayed window.location.reload().

G.16 AppConfig is NOT wrapped in a `<form onSubmit>`. Use a `<div>` with
     `type="button"` + onClick on the Save button. Reasons: (a) Grafana's
     outer plugin-settings page may or may not wrap us in its own <form>
     across versions; a nested submit is fragile, (b) keyboard-enter
     submission on an API-key field is a mild foot-gun (accidental submits).
     The Save button is the only action surface — keep it that way.

G.17 FieldConfig.DisplayNameFromDS is a pre-formatted final string — Grafana
     does NOT template-substitute it. Use FieldConfig.DisplayName with
     ${__field.labels.<name>} if you need interpolation, or bake the final
     string at emit time. We hit this when the CO₂ card legend rendered as
     literal "{{serial}}". See handleWirelessChannelUtil + handleSensorReadingsHistory
     + handleCameraAnalyticsOverview for canonical examples.

G.18 Native-timeseries frames must be one-frame-per-series with data.Labels
     on the value field. A single long-format frame with
     (ts, serial, metric, value) columns renders as an EMPTY chart — the
     timeseries viz can't infer series grouping from a mixed
     (time, string, number) shape. A client-side partitionByValues transform
     does NOT fix this (the residual string column stays on each partition
     and the viz still ignores it). Backend fix is the correct call.
     Canonical shapes: sensor_readings.go:handleSensorReadingsHistory,
     wireless.go:handleWirelessChannelUtil, appliance.go:handleDeviceUplinksLossLatency,
     camera.go:handleCameraAnalyticsOverview.

G.19 Query-handler signature takes Options as the 5th argument (LabelMode,
     PluginPathPrefix). The dispatcher threads plugin settings through
     Handle → runOne → handler. New handlers MUST accept the param even when
     unused (use `_ Options`). See pkg/plugin/query/dispatch.go — `Options`
     struct + `handlerFn` type. LabelMode is generic across every device
     family (sensor/MR/MS/MX/MV/MG); when it's "serial", per-device handlers
     skip the /devices lookup entirely (sensor_readings.go, wireless.go,
     appliance.go:handleDeviceUplinksLossLatency, camera.go).

G.20 Prefer server-side KPI aggregation over client-side filterByValue +
     reduce transform chains. Client transforms silently fall back to the
     wrong reducer in some Grafana versions; sensorAlertSummary /
     alertsOverview / applianceUplinksOverview / licensesOverview /
     apiRequestsOverview / clientsOverview exist because of this. Every
     future KPI row should follow the same "one wide frame, one field per
     KPI" pattern.

G.21 `q.Metrics[0]` overload. Several kinds repurpose the Metrics slot as a
     single-valued scalar to avoid churning the MerakiQuery wire shape:
       - alerts:                   severity filter
       - switchPortPacketCounters: port ID
       - licensesList:             state filter
       - cameraAnalyticsOverview:  objectType (person/vehicle)
       - cameraAnalyticsZoneHistory: zoneId; q.Metrics[1] optional objectType
     If a new kind needs a scalar beyond this list, consider adding a typed
     field to MerakiQuery instead — the overload is approaching its limit.

G.22 License struct name collision. pkg/meraki/organizations.go defines
     `OrgLicensingRef` (not `License`) for the nested "licensing" object in
     GET /organizations. `meraki.License` is the per-license record returned
     by GET /organizations/{orgId}/licenses (defined in pkg/meraki/insights.go).
     Linters or IDEs that auto-rename may collide; keep them distinct.

G.23 Plugin left-nav entries are driven server-side by `plugin.json::includes`.
     They cannot be hidden at runtime — even if the scene body tells us the
     selected org has zero devices of that family, the Grafana app shell has
     already mounted the nav item. `src/scene-helpers/familyGate.tsx` swaps
     the scene body for an "empty family" Alert instead, which is the best
     we can do without a plugin reload. A proper nav-entry hide-mode is a
     signed-release concern (would need a plugin-restart flow via the app
     config save path — see G.15 for why reloads aren't cheap).

7. API USAGE OPTIMISATION — RATE LIMITS, USER-AGENT, CACHING
================================================================================

Captures Meraki's documented optimisation levers and the Grafana-side caching

plugin less API-hungry".

Sources (verbatim in §7.6):
  - https://developer.cisco.com/meraki/api-v1/rate-limit/
  - https://developer.cisco.com/meraki/api-v1/user-agents-overview/

Cross-refs: §1.6 (rate limiter), §1.7 (cache TTLs), §G.1 (per-org limits),
§G.4 (freshness floor), §Q.1 / §Q.3 (distributed limiter / caching),
§4.1 P.9 (UA compliance action item).

--------------------------------------------------------------------------------
7.1  MERAKI-DOCUMENTED LIMITS (authoritative numbers)
--------------------------------------------------------------------------------

Organization (our primary constraint; rate limiter keys on orgID):
  - Steady-state: 10 requests per second per organization.
  - Burst:        10 requests in the first second; max 30 requests in 2s.
  - Shared across ALL API applications using the organization's API key.
  - Exceeding -> HTTP 429 + Retry-After header. Meraki's own guidance: "Pause
    for one or two seconds before making the next API call" and "increase wait
    times during high-volume periods".

Source IP (secondary):
  - 100 requests per second per source IP, shared across all clients using
    that address.
  - Relevant if this plugin runs with many orgs off one Grafana instance, OR
    if multiple tenants share an egress IP.

Current code posture (as of v0.3.0):
  - pkg/meraki/ratelimit.go: token bucket, 10 rps, burst 20, ±10% jitter,
    keyed on orgID. MATCHES the per-org cap.
  - No IP-level 100 rps ceiling modelled. See §7.4-G for an opt-in addition.

--------------------------------------------------------------------------------
7.2  USER-AGENT COMPLIANCE   (tracked as P.9 in §4.1)
--------------------------------------------------------------------------------

Required format per Meraki docs:
  "ApplicationName VendorName"
or with version:
  "ApplicationName/1.0 VendorName"

Rules (all required, not optional):
  - Omit all spaces, hyphens, and special characters from ApplicationName AND
    VendorName. So "Grafana-Meraki-App" is invalid (hyphens).
  - Forward slashes are reserved for version info only.
  - Must be consistent/correct across every request from the application.
  - "Do not omit the user agent string from any of your requests."

Why it matters beyond spec compliance: Meraki states the UA "helps your
customers and Meraki administrators monitor API interactions with their
environments and helps build their trust." Customers can see per-app traffic
attribution in their org's API Analytics dashboard. A compliant, versioned,
branded UA also makes this plugin findable in Cisco's ecosystem reports.

Current value: "Grafana-Meraki-App" (pkg/meraki/client.go:22,
pkg/plugin/app.go:128). NON-COMPLIANT — hyphens.

Target value: "GrafanaMerakiPlugin/<semver> rknightion" (flip vendor token
to "robknight" when Q.7 rename happens).

--------------------------------------------------------------------------------
7.3  MERAKI-RECOMMENDED API-EFFICIENCY LEVERS
--------------------------------------------------------------------------------

Four techniques for reducing request volume, and our status on each:

  A. "Use organization-wide API endpoints instead of per-device operations."
     Status: ALREADY FOLLOWED (§2.2 formalises this as a convention rule).
     Examples: GetOrganizationDevices over per-device loops; org-level
     switch/ports/statuses/bySwitch; org-level sensor readings;
     org-level appliance/uplink/statuses; org-level alerts/assurance.

  B. "Use Action Batches to bundle configuration requests."
     Status: NOT APPLICABLE. The plugin is read-only; Action Batches are
     POST-only. Revisit only if v1+ adds "configure from Grafana" flows.

  C. "Use getOrganizationConfigurationChanges instead of frequent polling."
     Endpoint: GET /organizations/{orgId}/configurationChanges
     Params: t0, t1, timespan. Cursor pagination (startingAfter/endingBefore).
     Use case: a "Change Log" scene showing who changed what, when.
     Status: PENDING (§4.2). TTL proposal: 5m. New files:
     pkg/meraki/configuration_changes.go + pkg/plugin/query/configuration_changes.go
     + a scene under src/pages/AuditLog/.

  D. "Replace repetitive polling with change-history endpoints like
      getOrganizationDevicesAvailabilitiesChangeHistory."
     Endpoint: GET /organizations/{orgId}/devices/availabilities/changeHistory
     Params: t0, t1, timespan, perPage (3-1000), startingAfter, endingBefore,
     networkIds, serials, productTypes.
     Status: PENDING (§4.2). TTL proposal: 30s. Additive to the existing
     DeviceAvailabilities kind; do NOT replace it.

  E. "Monitor actual usage via API Analytics in the Dashboard."
     Status: user-facing guidance; README addition after P.9 ships ("After
     deploying, watch dashboard.meraki.com -> your org -> API & Webhooks ->
     API requests — the plugin reports as GrafanaMerakiPlugin/<semver>
     rknightion once P.9 lands.").

--------------------------------------------------------------------------------
7.4  CACHING + REQUEST-HYGIENE IMPROVEMENTS
--------------------------------------------------------------------------------

Current state (pkg/meraki/cache.go):
  - TTLCache: golang-lru/v2 + per-entry expiry.
  - 2048-entry LRU shared across every endpoint, every org.
  - Per-kind TTLs enumerated in §1.7.
  - Keyed on sha256(orgID + path + sorted params).
  - Scope: per plugin-instance, in-memory. No Redis, no cross-replica share.

Improvements to consider, in priority order.

  A. Singleflight / request coalescing.    [DONE — v0.4.0, §18]
  B. Cache-key jitter on TTL.              [DONE — v0.4.0, §18]
  C. Stale-while-revalidate (SWR).         [DONE — v0.4.0, §18]
     Implemented as auto-derived grace = TTL/2 capped at 30m, which lines up
     with the per-kind proposals listed in the original §7.4-C spec.
  D. Negative caching of 404.              [DONE — v0.4.0, §18]
     Default 60s negative TTL; 401/403/5xx/412 deliberately NOT cached.
  E. Partitioned per-org cache.            [DONE — v0.4.0, §18]
     512 entries per org partition (up from a shared 2048 global LRU).
  F. External cache for multi-replica.     [DEFER — Q.3]
     Not planned for v0.x.
  G. Opt-in IP-level rate limiter.         [DONE — v0.4.0, §18]
     Off by default; wired via AppJsonData.enableIPLimiter.

--------------------------------------------------------------------------------
7.5  GRAFANA-SIDE CACHING INTERACTIONS
--------------------------------------------------------------------------------

Grafana's own caching layers (informational, not action items):

  - Grafana's backend plugin cache (data.Cache in the SDK) caches final
    data.Frames, not raw responses. Complementary to our TTLCache, not a
    replacement — we cache BEFORE frame assembly so a single cache entry can
    feed different frame shapes. No change proposed.
  - Grafana query result caching (Enterprise) sits in FRONT of the plugin.
    When enabled, effective TTL becomes max(our_ttl, grafana_ttl). Document
    in README once §7.4 improvements ship so operators don't double-tune.
  - "Query result reuse via instance manager": NewApp() is per plugin
    instance; Dispose() on rebuild. Grafana may rebuild instances on setting
    changes, flushing our cache — that's EXPECTED (settings changes
    invalidate key hashes).

Shipped:
- User-Agent compliance (P.9). New file pkg/meraki/version.go holds the
  ClientVersion const + BuildUserAgent() helper. The UA string is now
  "GrafanaMerakiPlugin/<version> rknightion" (no hyphens, single forward

  attributes this plugin's traffic correctly. Bump ClientVersion alongside
  package.json + CHANGELOG at every release.

- Cache upgrades (§7.4-A/B/C/D/E, all in pkg/meraki/cache.go):
    • Per-org partitioning — map[orgID]*lru.Cache, 512 entries each
      (up from a 2048 shared global LRU). Prevents a big-org operator
      evicting a quiet small-org operator's entries.
    • TTL jitter (±10% default, configurable via TTLCacheConfig.JitterRatio)
      so parallel replicas don't stampede at expiry.
    • Stale-while-revalidate via a per-entry stale-grace window. Client
      auto-derives grace = TTL/2, capped at 30 m, which matches the
      per-kind proposals from the original §7.4-C spec.
    • Negative-404 caching via StoreNotFound (default 60 s; 401/403/5xx/412
      deliberately NOT cached — those can resolve on the very next call once
      the user fixes their config).
  The public Lookup/Store API is new; legacy Get/Set are kept as
  backward-compat wrappers so existing tests and any external caller still
  work.

- Singleflight request coalescing (§7.4-A). golang.org/x/sync promoted from
  indirect to direct dep. Client.Get / Client.GetAll coalesce concurrent
  cache-miss round-trips on the CacheKey — N panels querying the same
  endpoint fan in to 1 HTTP request. Stale-hit path kicks an async refresh
  on a detached context.Background with 30 s / 60 s timeout; the same sf
  Group absorbs both direct and refresh calls so stampedes are impossible.

- Opt-in IP-level rate limiter (§7.4-G). A second RateLimiter instance
  keyed on "ip" (100 rps, 200 burst), acquired before the per-org bucket.
