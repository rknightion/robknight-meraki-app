Note on the "Early Access API" the user asked about:
Meraki's Early Access API is an opt-in program per-organization, not a
separate base URL or API version. Endpoints live under the same
`/api/v1/...` path and are dispatched to the caller's account based on the
opt-ins on `/organizations/{orgId}/earlyAccess/features/optIns`. Many of
the "new" endpoints flagged below (wireless clients overview, alert
overview/byNetwork, port overview, packet loss byNetwork, etc.) started
life in the EA program and are now fully GA — we do not need to add
early-access opt-in plumbing to call them. If we ever reach for a v1.52+
endpoint that is still EA-gated, we can add a CheckHealth probe that
warns when the feature isn't opted-in, but that's not required for any
recommendation below.

§1.1 MV camera analytics (five endpoints) — deprecated March 2024
------------------------------------------------------------------------------
[DONE — commit 55c6a12] Four deprecated wrappers removed; replaced with
GetOrganizationCameraBoundariesAreasByDevice / …LinesByDevice /
…DetectionsHistoryByBoundaryByInterval. KindCameraBoundaryAreas /
…Lines / …DetectionsHistory added. Cameras detail page: Analytics tab now
renders boundary detection counts; Zones tab renamed to Boundaries with a
merged areas+lines table. Per-zone TTL entry removed from timerange.go.
------------------------------------------------------------------------------
Currently used, all confirmed DEPRECATED per
https://developer.cisco.com/meraki/api-v1/deprecated-operations:

  - GET devices/{serial}/camera/analytics/live
  - GET devices/{serial}/camera/analytics/overview
  - GET devices/{serial}/camera/analytics/zones
  - GET devices/{serial}/camera/analytics/zones/{zoneId}/history
  - (we don't use `analytics/recent` but it's deprecated too)

Source files calling them:
  - pkg/meraki/camera.go  (GetDeviceCameraAnalyticsOverview @ L112,
                           GetDeviceCameraAnalyticsLive @ L141,
                           GetDeviceCameraAnalyticsZones @ L166,
                           GetDeviceCameraAnalyticsZoneHistory @ L195)
  - pkg/plugin/query/ — wherever the Cameras page dispatches these kinds
    (search for `CameraAnalytics` in pkg/plugin/query/ and src/pages/Cameras/)
  - pkg/meraki/timerange.go L193 KnownEndpointRanges entry for the `overview`
    variant
  - src/pages/Cameras/panels.ts — panels consuming these query kinds

Replacement (Meraki's prescribed path): the new **"boundaries"** model.

  A boundary is a user-configured area or line on a camera. The new
  endpoints return object-detection counts (person / vehicle) aggregated
  by boundary + interval.

  New endpoints to adopt:
    - GET organizations/{organizationId}/camera/boundaries/areas/byDevice
        → list area boundaries for cameras in the org (replaces `zones`)
    - GET organizations/{organizationId}/camera/boundaries/lines/byDevice
        → list line boundaries for cameras in the org
    - GET organizations/{organizationId}/camera/detections/history/
          byBoundary/byInterval
        → object-detection counts (person/vehicle) per boundary over time;
          replaces `analytics/zones/{zoneId}/history` and is also the
          de-facto replacement for `analytics/overview`
    - GET devices/{serial}/camera/sense/objectDetectionModels
        → list which object types the camera can detect (model-specific)

Why the change is non-trivial:
  - The data shape is fundamentally different. Old endpoints returned
    "zones" (per-camera) with aggregate `entrances`/`averageCount`. New
    endpoints return "boundaries" (per-camera) with in/out counts per
    object type (person/vehicle/etc.) per interval. Dashboards will need
    re-designing rather than field-for-field swapping.
  - The new `/camera/detections/history/byBoundary/byInterval` endpoint
    is org-level: a single call returns data for multiple boundaries
    across multiple cameras — a big efficiency win over the N-camera
    fan-out we do today.
  - Cameras without boundaries configured in Dashboard return empty
    results. We should surface this in the Camera panel as an empty-state
    with a link to https://documentation.meraki.com/MV/.../Area_and_Line_Crossing
    rather than showing zero series.

Implementation steps:
  1. Add wrappers in pkg/meraki/camera.go:
      - GetOrganizationCameraBoundariesAreasByDevice(ctx, orgID, ttl)
      - GetOrganizationCameraBoundariesLinesByDevice(ctx, orgID, ttl)
      - GetOrganizationCameraDetectionsHistoryByBoundaryByInterval(
          ctx, orgID, params {t0,t1,boundaryIds,boundaryTypes,
          perPage,ranges}, ttl)
      - (optional) GetDeviceCameraSenseObjectDetectionModels(ctx, serial, ttl)
  2. Add corresponding query kinds + handlers in pkg/plugin/query/
     (follow the pattern in pkg/plugin/query/CLAUDE.md — kind constant
     in src/datasource/types.ts, dispatch case, Go handler, frontend
     scene panel).
  3. Add a KnownEndpointRanges entry for
     `organizations/{organizationId}/camera/detections/history/
      byBoundary/byInterval` in pkg/meraki/timerange.go (check docs for
     MaxTimespan — typically 31 days — and allowed intervals).
  4. Delete:
       - pkg/meraki/camera.go funcs GetDeviceCameraAnalyticsOverview/Live/
         Zones/ZoneHistory + their request types
       - timerange.go entry for `devices/{serial}/camera/analytics/overview`
       - the matching query kinds in pkg/plugin/query/ and
         src/datasource/types.ts
       - panels in src/pages/Cameras/panels.ts that consumed them
  5. Replace those panels with boundary-based equivalents (in/out per
     person+vehicle stacked by boundary + a by-camera overview stat).
  6. Update src/pages/Cameras/panels.test.ts fixtures.
  7. Run: `npm run typecheck`, `npm run test:ci`, `go test ./pkg/...`.

Testing:
  - Requires a camera with at least one area + one line boundary in a
    real org to validate shape. Suggest plumbing a stub in
    pkg/plugin/query/dispatch_test.go that serves canned JSON from
    Meraki's docs example payload.

Priority: HIGH. Meraki has not yet announced a sunset date for the
deprecated endpoints but has said "replaced by newer 'boundaries'
operations" — they will remove them eventually. Also, the boundaries
endpoints return richer data, so we get a product win alongside the fix.

§2.1 Per-AP client list → org-level AP client counts
------------------------------------------------------------------------------
[DONE — commit 540fe1f] GetOrganizationWirelessClientsOverviewByDevice wrapper
+ KindWirelessApClientCounts + handleWirelessApClientCounts + apClientsByDeviceTable()
panel wired into AccessPoints overview. Per-AP fan-out retained as a
drill-down for the client-list panel.
------------------------------------------------------------------------------
Currently used:
  - GET devices/{serial}/clients
    (pkg/meraki/wireless.go L266, wrapper GetApClients; used by the
     AccessPoints page in src/pages/AccessPoints/panels.ts)

Status: VALID (reaffirmed in v1.44.0 with an added `properties` field).
Problem: for an N-AP network we have to fan out N requests to build
"clients per AP" panels. That's expensive on rate-limit budget and
serialises badly at scale. Meraki added an org-level one-shot endpoint
specifically for this visualisation.

Proposed replacement / augmentation:
  - GET organizations/{organizationId}/wireless/clients/overview/byDevice
    (added v1.51.1, GA)
    Returns `{items:[{network:{id},serial,counts:{byStatus:{online}}}]}`
    — one API call returns the current online-client count for every AP
    in the org.

Rationale:
  - 1 API call replaces N calls for the "clients per AP" panel.
  - Lower latency, far lower rate-limit pressure.
  - Org-scoped payload — pairs nicely with our existing
    `organizations/{organizationId}/devices` call for AP metadata.

Caveats:
  - This endpoint only returns client *counts*, not the full client
    list (MAC, SSID, usage, etc.). Panels that need per-client detail
    (e.g. the current "Top clients on AP" panel) must keep using
    `devices/{serial}/clients` — or switch to the org-level
    `organizations/{organizationId}/clients/search` / `clients/overview`
    for org-wide client aggregates.
  - Use it for count-only panels. Keep `devices/{serial}/clients`
    available for drill-down views but plumb it through a dashboard
    variable so we only fetch when the user selects a specific AP.

Implementation steps:
  1. Add GetOrganizationWirelessClientsOverviewByDevice wrapper in
     pkg/meraki/wireless.go (paginated with Link header).
  2. New query kind (e.g. `wirelessApClientCounts`) in
     src/datasource/types.ts + pkg/plugin/query/switches.go-style
     handler under pkg/plugin/query/.
  3. Rewrite the "clients per AP" panel in src/pages/AccessPoints/panels.ts
     to consume the new kind; demote the per-AP `devices/{serial}/clients`
     call to a row-drill-down only (fetched on row click).
  4. Keep timerange wrapper for the new endpoint — it's a snapshot, no
     KnownEndpointRanges entry needed.
  5. Add unit test in wireless_test.go (if absent, create) mocking the
     v1.51.1 example payload.

Priority: HIGH. Near-term efficiency win; no data-model migration cost.


§2.2 Per-device uplink loss/latency with longer history
------------------------------------------------------------------------------
[DONE — commit 591fedb] Augmenting (not replacing) the existing 5-minute
org-wide snapshot. GetDeviceUplinksLossAndLatencyHistory wrapper, 31-day
MaxTimespan entry in timerange.go, KindDeviceUplinksLossLatencyHistory
handler, uplinkLossLatencyHistoryTimeseries() panel on applianceUplinksScene
using a new $mx single-select variable.
------------------------------------------------------------------------------
Currently used:
  - GET organizations/{organizationId}/devices/uplinksLossAndLatency
    (pkg/meraki/appliance.go L368, wrapper GetOrganizationDevicesUplinksLossAndLatency,
     timerange.go L145 entry: **MaxTimespan = 5 minutes** — anything longer
     is a 400)

Status: VALID but fundamentally a "last 5 minutes" snapshot. Time-series
panels on this are always stuck at 5 min regardless of what the user
selects in the dashboard time picker.

Proposed augmentation (not replacement):
  - GET devices/{serial}/uplinks/lossAndLatencyHistory
    Returns per-device loss + latency time-series for up to 31 days,
    with configurable `interval` (60, 600, 3600, 86400 s depending on
    timespan).

Rationale:
  - Allows panels like "MX WAN latency over the last 24h" or
    "packet loss trend for device XYZ this week" — today we cannot build
    these at all.
  - Pairs with the existing org-level snapshot for a two-tier view:
    org overview for "who's hot right now" + per-device drill-down
    with full history.

Caveats:
  - Per-device endpoint → fan-out risk. Restrict via Grafana variable
    (e.g. template a single device serial) rather than auto-fetching
    all MXs on an org.

Implementation steps:
  1. Add GetDeviceUplinksLossAndLatencyHistory wrapper in
     pkg/meraki/appliance.go.
  2. Add KnownEndpointRanges entry in timerange.go:
     `devices/{serial}/uplinks/lossAndLatencyHistory`:
       MaxTimespan: 31 * 24h,
       AllowedResolutions: [60, 600, 3600, 86400] (verify via docs).
  3. Add query kind in dispatch layer; require a `serial` variable in
     the scene.
  4. Add a panel on the MX / Appliance page: "Uplink latency history"
     with a device-selector variable at the top.

Priority: MEDIUM-HIGH. Unblocks a real user need (historical WAN
visibility) that the current snapshot cannot answer.

==============================================================================
§6  Early Access API — summary for reviewer
==============================================================================

The Meraki Early Access API is not a different base URL, API version, or
OpenAPI spec. It's a per-org opt-in gate that enables certain beta
endpoints (or new response fields on existing endpoints) for participating
organizations. All paths recommended in §2 and §3 are GA and do not
require opting in.

If we later want to use a specifically EA-gated endpoint, the pattern is:

  1. Call GET /organizations/{organizationId}/earlyAccess/features to
     list what's offered.
  2. Call POST /organizations/{organizationId}/earlyAccess/features/
       optIns with {featureId} to opt-in (requires write scope on
     the key — the plugin currently uses read-only scopes as much as
     possible; do NOT auto-opt-in an org from this plugin, just warn).
  3. Once opted in, the endpoint appears normally on the same
     `/api/v1/...` path.

In the absence of a specific EA-gated endpoint we actually want, no work
is required here. This section exists so future contributors do not
spend time looking for a phantom "early-access base URL" that does not
exist.
